# 部署文档

本文档面向服务器部署与上线运维，基于当前项目实际代码与目录编写。

---

## 1. 部署前准备

1. 确认部署方式：
   - Docker Compose（推荐）
   - Node.js 直接运行
2. 准备管理员密码：`ADMIN_PASSWORD`
3. 准备数据目录备份策略（`./data`）
4. 如需 webhook 推送，确认目标域名与网络策略

---

## 2. 运行环境要求

### 方案 A：Docker 部署（推荐）
- Docker Engine + Docker Compose
- 可写磁盘目录用于挂载数据卷

### 方案 B：Node 直接部署
- Node.js 20+
- pnpm（建议 `corepack enable`）
- Linux/macOS/Windows 均可（生产推荐 Linux）

---

## 3. 依赖安装步骤

### Node 直接部署

```bash
corepack enable
pnpm install
```

> 首次部署必须先构建前端，再启动后端。

---

## 4. 前端打包与部署步骤

项目前端构建产物为 `web/dist`，由后端静态托管。

```bash
pnpm build:web
```

后端会读取 `web/dist`，若缺失会提示 `web build not found`。

---

## 5. 后端部署步骤

### 方式 A：Docker Compose

1. 在项目根目录创建 `.env`（示例见第 6 节）
2. 执行：

```bash
docker compose up -d --build
```

3. 查看日志：

```bash
docker compose logs -f
```

4. 停止服务：

```bash
docker compose down
```

### 方式 B：Node 进程

```bash
pnpm build:web
ADMIN_PASSWORD='请替换为强密码' pnpm dev:core
```

或：

```bash
pnpm -C core start
```

> 建议使用 systemd / PM2 托管进程（需人工确认具体运维体系）。

---

## 6. 配置文件 / 环境变量配置方法

推荐根目录 `.env`（Docker 或运维平台注入均可）：

```dotenv
ADMIN_PASSWORD=请替换为强密码
ADMIN_PORT=3000
ADMIN_TOKEN_TTL_HOURS=24
ADMIN_CORS_ORIGINS=http://localhost:3000,http://127.0.0.1:3000
ADMIN_SESSION_STORE=memory

# 账号 Code 加密密钥（推荐设置）
ACCOUNT_CODE_KEY=64位十六进制密钥
# 可选，多密钥解密兼容（逗号分隔）
# ACCOUNT_CODE_KEYS=key1,key2

# 可选，仅单机开发建议启用
# ACCOUNT_SECRET_ALLOW_LOCAL=1

# 可选，限制 webhook 目标域名白名单
# WEBHOOK_ALLOWED_HOSTS=example.com,api.example.com

# 可选，日志级别
# LOG_LEVEL=info

# 可选，worker 模式（thread/fork）
# FARM_RUNTIME_MODE=thread
```

说明：
- `ADMIN_PASSWORD` 必填；未设置会导致登录不可用。
- `ADMIN_SESSION_STORE` 默认 `memory`，重启后会话失效；若需落盘可改 `file`。
- 不建议手工设置 `FARM_WORKER`、`FARM_ACCOUNT_ID`（内部变量）。

当前项目未使用数据库，正式持久化在 `data/`：

- `accounts.json`

## 8. 反向代理 / 端口 / 静态资源处理

## 默认端口
- 后端监听：`0.0.0.0:${ADMIN_PORT}`，默认 `3000`
- Docker 默认映射：`3000:3000`

## 静态资源
- 后端托管 `web/dist`
- 路由兜底：非 `/api` 与 `/game-config` 请求返回 `index.html`

## WebSocket
- Socket.io 路径：`/socket.io`
- 反向代理需开启 WebSocket 转发

## Nginx 示例（可选）

```nginx
server {
    listen 80;
    server_name your.domain;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /socket.io/ {
        proxy_pass http://127.0.0.1:3000/socket.io/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
    }
}
```

> 如使用 HTTPS、WAF、网关鉴权等，请按你的基础设施补充（需人工确认）。

## 9. 生产环境部署建议

1. 强制设置高强度 `ADMIN_PASSWORD`
2. 使用固定 `ACCOUNT_CODE_KEY`，避免密钥漂移
3. 如需要多设备保持登录，`ADMIN_SESSION_STORE=file` 并保护 `data/` 目录权限
4. 通过 `ADMIN_CORS_ORIGINS` 精确限制来源
5. webhook 场景建议设置 `WEBHOOK_ALLOWED_HOSTS`
6. 对 `data/` 目录做定期备份
7. 通过进程管理器托管（systemd/PM2，需人工确认）

## 10. 常见部署问题与排查

### 10.1 页面可访问但接口 401/403
- 检查 `ADMIN_PASSWORD` 是否正确设置
- 检查 `ADMIN_CORS_ORIGINS` 与实际访问源是否匹配
- 检查反向代理是否透传必要 Header

### 10.2 启动后提示 web build not found
- 未执行 `pnpm build:web`
- 部署路径中缺失 `web/dist`

### 10.3 重启后登录失效
- 默认 `ADMIN_SESSION_STORE=memory`
- 若需持久化改为 `file`

### 10.4 推送测试失败
- 检查 webhook 地址、白名单、网络出口
- 检查 DNS 与防火墙策略

### 10.5 data 目录无写权限
- 容器挂载目录权限不足
- 非容器模式下进程用户无写权限

## 12. 回滚建议

1. 保留上一个可运行版本代码/镜像
2. 升级前备份 `data/` 目录
3. 发生异常时：
   - 回滚代码/镜像
   - 恢复 `data/` 备份
   - 重启服务并验证 `/api/ping`、登录、账号任务

> 若跨大版本包含数据结构变化，请先在预发布环境演练回滚（需人工确认）。
